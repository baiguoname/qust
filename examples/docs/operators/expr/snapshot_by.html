使用场景

用于构造表达式链路中的核心计算、上下文切换、调试或元数据操作。

当你需要 expr.snapshot_by 这种语义时，优先使用这个算子，而不是在 Python 里提前把数据取出来手写循环；这样才能保留流式状态、参数联动、Wasm 路径和 Monitor 重算能力。

为什么这样设计

Qust 把这一类能力做成表达式节点，是为了把“用户想算什么”保留下来。只要语义还在表达式树里，后续就可以统一处理分组上下文、窗口状态、参数联动、Wasm 执行和 Monitor 可视化。

实际收益是：代码更短，但语义更完整；后续计算、画图、调参、保存和回调都能复用同一棵表达式树。

注意事项

- `by` 为空：`snapshot_by(by, ...) 至少需要 1 列标签`
- `keep_labels < 1`：`snapshot_by keep_labels 必须 >= 1`
- `start_calc_n < 1`：`snapshot_by start_calc_n 必须 >= 1`
- `by` 行数与输入行数不一致：会触发长度检查错误。
- 若下游链路包含不满足“可回放一致性”的自定义逻辑，结果可能偏离预期，应先做最小链路验证。

**上下文算子（高级）**，按标签维护“执行器快照基线”，并在标签回流/乱序时做**状态一致性重算**。

核心定义 ✅
-----------
`snapshot_by` 解决的是“流式分块输入 + 标签可能回流（同一标签未来又来新行）”场景下，
下游状态算子（如 `over`、`rolling`、`expanding`、`batch_ta.rank` 等）的可重复、可收敛计算问题。

其核心机制不是简单 append，而是：
1. 维护一个“快照基线执行器（snapshot exec）”；
2. 维护从快照基线之后的历史数据缓存；
3. 当检测到需要回流重算时，从快照基线复制执行器并对缓存历史重放；
4. 用重放后的新执行器替换 live 执行器，保证后续状态一致。

与 `batch(...)` 的关系与区别 🧠
-------------------------------
- `batch(...)`：强调 batch packet 分发语义（update/retract 路径）。
- `snapshot_by(...)`：强调“快照 + 回放”语义，不依赖 batch packet 注入。
- 如果你的目标是“先保证结果一致性，再逐步优化 retract 性能”，通常 `snapshot_by` 更稳妥。

输入 📥
-------
- 函数签名：
  `snapshot_by(*by: ExprInput, keep_labels: int = 64, start_calc_n: int = 1, need_active: bool = False)`
- `by`：
  - 一个或多个标签列表达式（至少 1 列），常见如 `"datetime"`、`("date", "code")`。
  - 每次 `calc_data` 输入中，`by` 计算后的行数必须与当前输入行数一致。
- `keep_labels`：
  - 标签缓存上限（必须 >= 1）。
  - 超出后会淘汰最旧标签，并将当前 live 执行器固化为新的快照基线。
- `start_calc_n`：
  - 每个标签累计到第 `n` 行后才“激活输出”（必须 >= 1）。
  - 典型用途：避免标签样本不足时输出不稳定信号。
- `need_active`：
  - `False`：激活前历史不补发（输出更省内存，吞吐更高）。
  - `True`：一旦激活，补齐该标签激活前历史（更完整）。

返回 📤
-------
- 返回类型：`Expr`
- 返回语义：将当前表达式包装为带 `snapshot_by` 上下文的新表达式，后续可继续链式调用。

状态机与执行流程（详细）⚙️
---------------------------
每次收到新分块数据（一次 `calc_data(...)` 调用）时：

1. 计算标签
   - 对 `by` 计算当前分块标签，映射成内部 group id。
   - 若 `by` 为空或标签长度与行数不一致，直接报错。

2. 计算“是否需要重算”
   - 如果当前分块出现了“已存在过的标签”，标记 `recalc_needed=True`。
   - 新标签仅走增量路径，不触发全量回放。

3. 维护缓存与激活窗口
   - 记录当前分块原始数据、标签、标签代际信息到历史缓存。
   - 根据 `start_calc_n/need_active` 计算 active 行集合。

4. 两条执行路径
   - 增量路径（`recalc_needed=False`）：
     直接把当前分块送入 live 执行器计算。
   - 回放路径（`recalc_needed=True`）：
     从快照基线复制执行器，重放“快照之后历史”，再替换 live 执行器。

5. 标签淘汰与基线推进
   - 若标签数超过 `keep_labels`，淘汰最旧标签；
   - 同时把当前 live 执行器快照成新的基线，并清理历史缓存，避免无界增长。

输出形态说明（很重要）📌
------------------------
- 在纯增量路径下，输出通常对应当前分块（可能受 `start_calc_n/need_active` 影响被裁剪）。
- 在回放路径下，可能返回“重放后可见范围”的结果，而不只是当前分块。
  这属于设计语义，用于确保状态一致，不是 bug。

适用场景 🧭
-----------
- 数据按块到来，同一标签后续可能补数据（如日内截面补齐、延迟到达、交易时段拼接）。
- 下游存在状态算子，且你不能接受“只 append 导致的历史偏差”。
- 需要在工程复杂度和一致性之间优先选择一致性。

不建议场景 ❌
-------------
- 标签永不回流，且追求极限吞吐：可优先用纯增量路径。
- 历史极长且 `keep_labels` 过大但机器内存有限：需评估内存占用。

---------------

>>> out = qs.select(e).calc_data(data)

参数调优建议 🎛️
---------------
- `keep_labels`：
  - 太小：频繁淘汰，重放窗口变短，可能导致你观察到更频繁“基线重置”。
  - 太大：内存增大，回放开销上升。
- `start_calc_n`：
  - 对新标签冷启动噪声大时可适当增大。
- `need_active`：
  - 回测/审计要完整轨迹时设 `True`；
  - 在线低延迟只关心最新时点时设 `False`。

可能出错的地方 ⚠️
------------------
- `by` 为空：`snapshot_by(by, ...) 至少需要 1 列标签`
- `keep_labels < 1`：`snapshot_by keep_labels 必须 >= 1`
- `start_calc_n < 1`：`snapshot_by start_calc_n 必须 >= 1`
- `by` 行数与输入行数不一致：会触发长度检查错误。
- 若下游链路包含不满足“可回放一致性”的自定义逻辑，结果可能偏离预期，应先做最小链路验证。

性能与资源注意事项 🚨
---------------------
- `snapshot_by` 通过“快照 + 重放”保证一致性，代价是回流时会有额外重算成本。
- 历史缓存与标签缓存都占内存，参数设置应结合数据规模压测。
- 对极高吞吐实时场景，建议先用小规模数据验证参数，再放量。
